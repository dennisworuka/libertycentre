# Decisions Log

Records ambiguous calls made during the build, per `CLAUDE.md`: "When genuinely ambiguous, choose the more secure and more accessible option and note the decision here."

## Phase 1 — Foundations

- **Local PHP version is 8.3.32, not 8.5.** Ubuntu 26.04's own repos only ship PHP 8.5; PHP 8.3 (matching the production cPanel target) was installed from `packages.sury.org` instead. Composer's `platform.php` is not pinned, so keep local PHP at 8.3 deliberately — don't let a distro upgrade silently move dev onto 8.5 while production stays on 8.3.
- **Local dev toolchain lives in WSL Ubuntu, project files stay on the Windows filesystem (`D:\Liberty` / `/mnt/d/Liberty`).** The Windows host has no PHP/Composer/MySQL. WSL was chosen over Docker/Sail because Docker Desktop's daemon needed a manual GUI start this session; WSL let the agent install and drive PHP/Composer/MariaDB directly without a human in the loop for every command. Run backend commands (`composer`, `artisan`, `pest`, `mysql`) via `wsl -d Ubuntu -- bash -lc "cd /mnt/d/Liberty && ..."`; run `npm`/Vite from Windows directly since Node already exists there and file-watching across the WSL/Windows filesystem boundary is unreliable.
- **Test database:** using Laravel's default sqlite `:memory:` for the Pest suite (the framework default) rather than the local MariaDB `liberty_centre_testing` database created during environment setup. Faster and standard practice; MariaDB-specific behaviour isn't exercised by the domain code in Phase 1. Revisit if a later phase relies on MySQL-only SQL (e.g. fulltext search).
- **2FA login is a custom two-step Filament page (`App\Filament\Auth\Login`), not a Filament plugin.** No off-the-shelf Filament 3 2FA plugin was in the required package list, and the spec's "no exceptions" wording implied full control over the flow. Step 1 checks password only; step 2 requires a TOTP or recovery code before `Auth::guard()->login()` is ever called — no authenticated session/cookie exists until the code passes. "Remember me" was dropped entirely: a persistent remember cookie set before 2FA passes would undermine the whole point of mandatory 2FA, and re-checking TOTP every session is the stricter, intended reading of "no exceptions."
- **2FA is provisioned at account-creation time, not via self-service enrolment**, via a `User::booted()` `creating` hook that generates a secret + 8 recovery codes + `two_factor_confirmed_at` whenever a new user is saved without one already set. This makes "every account has 2FA, no exceptions" a structural guarantee (holds for the seeder, the Filament UserResource, tinker, factories — anything that creates a `User` row) rather than a UI flow that could be skipped. The secret/QR/codes are shown once, via a persistent Filament notification on `CreateUser::afterCreate()` (mirroring what `SuperAdminUserSeeder` prints to the console).
- **Per-account progressive lockout (`App\Services\Auth\LoginThrottle`) is a second, independent layer from Filament's built-in per-IP rate limit** (`WithRateLimiting`, 5/min, already inherited from `Filament\Pages\Auth\Login`). The spec asked for both "5 attempts/min/IP" (Filament's default, unmodified) and "per-account lockout with progressive delay" (ours, keyed by email, exponential backoff capped at 60 minutes). They're deliberately independent so a distributed attack against one account still locks out even when spread across many IPs.
- **`SESSION_LIFETIME` is 30 minutes application-wide**, not just for `/admin`. Laravel's session lifetime is a single app-wide config value, and there's no first-party per-route-group session-lifetime mechanism without custom middleware; since the public site has no long-lived session use case yet (Phase 4's referral wizard should comfortably fit in 30 minutes), applying the admin's "30-minute idle timeout" requirement globally is the simplest correct implementation, not a workaround. Revisit only if a later phase needs a longer-lived public session.
- **Policy model-parameter types must be `mixed`, not omitted, in `BasePolicy`.** PHP's parameter-variance rules reject a concrete policy narrowing an *untyped* parent parameter (e.g. `view(User $user, $model)`) to a specific class type — it only allows narrowing when the parent parameter is explicitly typed `mixed`. Every abstract method on `BasePolicy` uses `mixed $model = null` for exactly this reason; don't "simplify" it back to an untyped param.
- **`/public/build` was removed from `.gitignore`.** Laravel's default scaffold ignores it (assumes the server runs its own `npm run build`), but CLAUDE.md's hosting constraint is explicit: "compiled assets are committed for cPanel deployment (no Node on the server)." If this line ever creeps back in, the site ships with no CSS/JS on a fresh cPanel deploy.

## Phase 3 — Public Frontend

- **Shared site data (settings + navigation) is delivered via a `ShareSiteData` middleware calling `View::share()`, not a `View::composer()` bound to `layouts.app`.** A composer scoped to one view name only populates that specific view instance — it does not reach the data of a child template's own `@section('content')` block, because Blade renders that section content in the child's own data scope *before* the parent layout is even instantiated. This surfaced as "Undefined variable `$siteOrganisation`" on every page and blocked the whole template layer. `View::share()` puts the data on the shared view-data bag that every view (parent or child) can read, and it's registered on the `web` middleware group only so Filament's admin panel — which uses its own explicit middleware list — never pays for the settings/navigation queries.
- **`spatie/laravel-settings` cannot resolve PHPStan-only shaped-array docblocks at runtime.** Its `PropertyReflector` parses `@var` tags with `phpDocumentor\Reflection\TypeResolver`, which throws (`Unexpected token`) on a PHPStan shape like `array<int, array{question: string, rating: string}>`, and still throws building the cast on a simplified two-level generic. Fix: give the property a bare `array` type and document the shape with `@phpstan-var` instead of `@var` — spatie's own docblock parser only regex-matches the literal string `@var `, so `@phpstan-var` is invisible to it while Larastan still picks it up. Any future Settings class property needing a non-trivial array shape should use this same `@phpstan-var` pattern rather than `@var`.
- **Skipped a full Playwright + axe-core E2E setup for this phase, in favour of Pest HTTP-level feature tests** (`tests/Feature/Public/*`) asserting status codes, key content, and the branded-404 fallback across every route. Phase 7 explicitly owns axe-core accessibility CI; standing up a second, overlapping E2E harness now would be premature.
- **Home page trust-bar stats (40+ people supported / 12 years / 24-7 support) are hardcoded in `pages/home.blade.php`**, not modelled as `OrganisationSettings` fields. They're marketing copy, not operational configuration nobody else needs to edit independently of a template change; extending the settings schema for three numbers not asked for in the spec would be speculative.
- **Official brand palette applied 2026-07-08.** `_variables.scss` now defines `$primary: #B14040` (red), `$secondary: #C46945` (orange), `$success: #78A345` (green), `$light: #FAFAFA`, `$dark: #2D2D2D`, `$body-color: #333333` — replacing the placeholder teal "Steady Hands" palette used to build out Phase 3's templates. Existing design-system role variables (`$ink`, `$surface`, `$mist`, `$amber`) were kept as aliases onto the new palette (`$ink → $dark`, `$surface → $light`, `$amber → $secondary`, `$mist` computed as a 10% dark/white mix) rather than rewriting every Blade/SCSS reference, so this was a single-point-of-truth swap in one file. `$danger` (`#B3261E`) was left unchanged — no danger color was specified, and it's deliberately distinct from the new brand red so error states don't read as "primary." CLAUDE.md's design-system section was updated to match.
