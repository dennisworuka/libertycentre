# Decisions Log

Records ambiguous calls made during the build, per `CLAUDE.md`: "When genuinely ambiguous, choose the more secure and more accessible option and note the decision here."

## Phase 1 — Foundations

- **Local PHP version is 8.3.32, not 8.5.** Ubuntu 26.04's own repos only ship PHP 8.5; PHP 8.3 (matching the production cPanel target) was installed from `packages.sury.org` instead. Composer's `platform.php` is not pinned, so keep local PHP at 8.3 deliberately — don't let a distro upgrade silently move dev onto 8.5 while production stays on 8.3.
- **Local dev toolchain lives in WSL Ubuntu, project files stay on the Windows filesystem (`D:\Liberty` / `/mnt/d/Liberty`).** The Windows host has no PHP/Composer/MySQL. WSL was chosen over Docker/Sail because Docker Desktop's daemon needed a manual GUI start this session; WSL let the agent install and drive PHP/Composer/MariaDB directly without a human in the loop for every command. Run backend commands (`composer`, `artisan`, `pest`, `mysql`) via `wsl -d Ubuntu -- bash -lc "cd /mnt/d/Liberty && ..."`; run `npm`/Vite from Windows directly since Node already exists there and file-watching across the WSL/Windows filesystem boundary is unreliable.
- **Test database:** using Laravel's default sqlite `:memory:` for the Pest suite (the framework default) rather than the local MariaDB `liberty_centre_testing` database created during environment setup. Faster and standard practice; MariaDB-specific behaviour isn't exercised by the domain code in Phase 1. Revisit if a later phase relies on MySQL-only SQL (e.g. fulltext search).
- **2FA login is a custom two-step Filament page (`App\Filament\Auth\Login`), not a Filament plugin.** No off-the-shelf Filament 3 2FA plugin was in the required package list, and the spec's "no exceptions" wording implied full control over the flow. Step 1 checks password only; step 2 requires a TOTP or recovery code before `Auth::guard()->login()` is ever called — no authenticated session/cookie exists until the code passes. "Remember me" was dropped entirely: a persistent remember cookie set before 2FA passes would undermine the whole point of mandatory 2FA, and re-checking TOTP every session is the stricter, intended reading of "no exceptions."
- **2FA is provisioned at account-creation time, not via self-service enrolment**, via a `User::booted()` `creating` hook that generates a secret + 8 recovery codes + `two_factor_confirmed_at` whenever a new user is saved without one already set. This makes "every account has 2FA, no exceptions" a structural guarantee (holds for the seeder, the Filament UserResource, tinker, factories — anything that creates a `User` row) rather than a UI flow that could be skipped. The secret/QR/codes are shown once, via a persistent Filament notification on `CreateUser::afterCreate()` (mirroring what `SuperAdminUserSeeder` prints to the console).
- **Per-account progressive lockout (`App\Services\Auth\LoginThrottle`) is a second, independent layer from Filament's built-in per-IP rate limit** (`WithRateLimiting`, 5/min, already inherited from `Filament\Pages\Auth\Login`). The spec asked for both "5 attempts/min/IP" (Filament's default, unmodified) and "per-account lockout with progressive delay" (ours, keyed by email, exponential backoff capped at 60 minutes). They're deliberately independent so a distributed attack against one account still locks out even when spread across many IPs.
- **`SESSION_LIFETIME` is 30 minutes application-wide**, not just for `/admin`. Laravel's session lifetime is a single app-wide config value, and there's no first-party per-route-group session-lifetime mechanism without custom middleware; since the public site has no long-lived session use case yet (Phase 4's referral wizard should comfortably fit in 30 minutes), applying the admin's "30-minute idle timeout" requirement globally is the simplest correct implementation, not a workaround. Revisit only if a later phase needs a longer-lived public session.
- **Policy model-parameter types must be `mixed`, not omitted, in `BasePolicy`.** PHP's parameter-variance rules reject a concrete policy narrowing an *untyped* parent parameter (e.g. `view(User $user, $model)`) to a specific class type — it only allows narrowing when the parent parameter is explicitly typed `mixed`. Every abstract method on `BasePolicy` uses `mixed $model = null` for exactly this reason; don't "simplify" it back to an untyped param.
- **`/public/build` was removed from `.gitignore`.** Laravel's default scaffold ignores it (assumes the server runs its own `npm run build`), but CLAUDE.md's hosting constraint is explicit: "compiled assets are committed for cPanel deployment (no Node on the server)." If this line ever creeps back in, the site ships with no CSS/JS on a fresh cPanel deploy.

## Phase 3 — Public Frontend

- **Shared site data (settings + navigation) is delivered via a `ShareSiteData` middleware calling `View::share()`, not a `View::composer()` bound to `layouts.app`.** A composer scoped to one view name only populates that specific view instance — it does not reach the data of a child template's own `@section('content')` block, because Blade renders that section content in the child's own data scope *before* the parent layout is even instantiated. This surfaced as "Undefined variable `$siteOrganisation`" on every page and blocked the whole template layer. `View::share()` puts the data on the shared view-data bag that every view (parent or child) can read, and it's registered on the `web` middleware group only so Filament's admin panel — which uses its own explicit middleware list — never pays for the settings/navigation queries.
- **`spatie/laravel-settings` cannot resolve PHPStan-only shaped-array docblocks at runtime.** Its `PropertyReflector` parses `@var` tags with `phpDocumentor\Reflection\TypeResolver`, which throws (`Unexpected token`) on a PHPStan shape like `array<int, array{question: string, rating: string}>`, and still throws building the cast on a simplified two-level generic. Fix: give the property a bare `array` type and document the shape with `@phpstan-var` instead of `@var` — spatie's own docblock parser only regex-matches the literal string `@var `, so `@phpstan-var` is invisible to it while Larastan still picks it up. Any future Settings class property needing a non-trivial array shape should use this same `@phpstan-var` pattern rather than `@var`. **Caveat found the hard way:** spatie's regex matches that literal substring *anywhere* in the docblock, including plain prose — a first attempt at this fix described the workaround in the comment itself ("no plain `@var` generic here"), and that description was enough to trip the same crash on every request (`class_implements(): Class Spatie\LaravelData\Data does not exist`), since `spatie/laravel-data` isn't installed. Never write the literal tag as text in a Settings property's docblock, even to explain why it's absent.
- **The generic `/{slug}` catch-all (and the `services/{slug}`/`news/{slug}` routes) shadow `Route::fallback`'s redirect lookup for any legacy single-segment path.** `Route::fallback` only fires when no route matches the URI *shape* — but `/{slug}` matches any single segment regardless of whether a Page exists, so `PageController::show()`'s `firstOrFail()` was 404ing before the Redirect table was ever checked, silently breaking every slug-changed redirect created by `HasSlug`. Fix: `PageController`/`ServiceController`/`PostController::show()` now do a plain `first()` and call a shared `Controller::redirectOrAbort()` helper (checks `Redirect`, then aborts) instead of `firstOrFail()`. Caught by re-running `RedirectFallbackTest` after Phase 3 landed the catch-all route — it's exactly the kind of interaction a per-file test run won't surface, since that test only existed in Phase 2's suite, not Phase 3's.
- **Branded error views (`errors/403` etc.) must not be rendered for Filament admin requests.** They `@extends('layouts.app')`, which needs settings shared by `ShareSiteData` — middleware deliberately scoped to the `web` group only, not the Filament panel. A 403 inside `/admin` was crashing instead of rendering. Fixed with an `$exceptions->render()` callback in `bootstrap/app.php` that returns a bare status response for any request under `config('admin.path')`, before Laravel's default `errors::` view lookup runs. Admin errors have no branded page as a result (Filament itself doesn't ship one either), which is an acceptable trade rather than making the entire public layout defensive against missing shared data.
- **Skipped a full Playwright + axe-core E2E setup for this phase, in favour of Pest HTTP-level feature tests** (`tests/Feature/Public/*`) asserting status codes, key content, and the branded-404 fallback across every route. Phase 7 explicitly owns axe-core accessibility CI; standing up a second, overlapping E2E harness now would be premature.
- **Home page trust-bar stats (40+ people supported / 12 years / 24-7 support) are hardcoded in `pages/home.blade.php`**, not modelled as `OrganisationSettings` fields. They're marketing copy, not operational configuration nobody else needs to edit independently of a template change; extending the settings schema for three numbers not asked for in the spec would be speculative.
- **Official brand palette applied 2026-07-08.** `_variables.scss` now defines `$primary: #B14040` (red), `$secondary: #C46945` (orange), `$success: #78A345` (green), `$light: #FAFAFA`, `$dark: #2D2D2D`, `$body-color: #333333` — replacing the placeholder teal "Steady Hands" palette used to build out Phase 3's templates. Existing design-system role variables (`$ink`, `$surface`, `$mist`, `$amber`) were kept as aliases onto the new palette (`$ink → $dark`, `$surface → $light`, `$amber → $secondary`, `$mist` computed as a 10% dark/white mix) rather than rewriting every Blade/SCSS reference, so this was a single-point-of-truth swap in one file. `$danger` (`#B3261E`) was left unchanged — no danger color was specified, and it's deliberately distinct from the new brand red so error states don't read as "primary." CLAUDE.md's design-system section was updated to match.
- **Working copy moved from `D:\Liberty` to WSL's native filesystem at `/root/liberty` (2026-07-08), after the site became barely usable.** Every file access from WSL to the Windows-mounted `/mnt/d/Liberty` goes through the 9p bridge; a bare `php artisan --version` — no HTTP, no database — took 30–47 seconds (only ~2.5s of that was CPU; the rest was I/O wait opening the ~1,000+ vendor files Laravel/Filament need to boot), and a full page load had taken as long as 9 minutes before failing outright. Tried opcache for the CLI SAPI, disabling `opcache.validate_timestamps`, `composer dump-autoload --optimize`, `filament:cache-components`, and a full `wsl --shutdown` — each helped marginally, none fixed it, because the bottleneck is directory/file-open latency, not compilation cost. Copying the working tree to native ext4 dropped the same bootstrap to ~2s and the full Pest suite from 471s to 9s, with no further opcache tuning needed (`validate_timestamps` was put back to its default of `1` so edited files are picked up live, as normal in dev). Node/npm was also installed natively in WSL (previously run from Windows against `D:\Liberty`) so Vite builds get the same benefit. `D:\Liberty` was left in place as a pre-migration backup, not deleted — see CLAUDE.md for the new edit/run workflow.

## Post-Phase-3 fix — Filament admin under the strict CSP

- **Filament 3.3.54 has no first-party CSP nonce support for its own inline `<script>`/`<style>` tags, and this silently broke the entire admin panel.** `SecurityHeaders` middleware applies the same strict `script-src 'self' 'nonce-...'` / `style-src 'self' 'nonce-...'` CSP (no `unsafe-inline`) globally, including `/admin/*`. Filament's base layout, sidebar, and unsaved-changes-alert views all ship hardcoded inline scripts/styles (dark-mode bootstrapping, `window.filamentData`, sidebar collapsed-group state, the active-sidebar-item scroll helper) with no nonce attribute and no config hook to add one — confirmed by grepping the whole `vendor/filament/*` and `vendor/livewire/livewire` trees for `nonce`. The browser silently dropped every one of them (no console-visible error surfaced to us), which broke Alpine/Livewire initialisation on `/admin/login` — the reported symptom was "the login form is not functional" (button click did nothing). This is exactly the class of bug automated HTTP-level Pest tests cannot catch, since Laravel's test client never runs a real JS engine or enforces CSP; only an actual browser does. Caught only once asked "is the admin ready?" and then actually inspecting rendered HTML for un-nonced tags.
- **Fix chosen: keep the CSP fully strict everywhere (including `/admin`) and make Filament's own output comply, rather than weakening the policy for admin routes.** The admin panel is the most security-sensitive surface in the app (PII, CV uploads, referral data in later phases), so a scoped `unsafe-inline` carve-out for `/admin` — the lower-effort fix — was rejected. Two parts:
  1. `AppServiceProvider::boot()` now calls `Vite::useCspNonce(app(Nonce::class)->value())` before any view renders. Livewire's own `FrontendAssets` mechanism (`@livewireScripts`/`@livewireStyles`, which Filament's layout uses internally) already falls back to `Vite::cspNonce()` when no explicit nonce option is passed — this was simply never wired up.
  2. Five Filament package views with hardcoded un-nonced inline tags are overridden byte-for-byte (Laravel's standard `resources/views/vendor/{package}/...` mechanism) with `@nonce` added to each `<script>`/`<style>` tag and nothing else changed: `vendor/filament-panels/components/layout/{base,index}.blade.php`, `vendor/filament-panels/components/sidebar/index.blade.php`, `vendor/filament-panels/components/{unsaved-action-changes-alert,page/unsaved-data-changes-alert}.blade.php`, and `vendor/filament/assets.blade.php` (from `filament/support`, hence the different top-level namespace). Found by grepping `vendor/filament/*/resources/views` for `<script>`/`<style>` without `@nonce`, cross-checked by rendering six representative authenticated admin pages (dashboard, resource list, create form, users, settings) through Pest's `actingAs()` and re-scanning the output — all clean.
- **These five view overrides are version-pinned to filament/filament and filament/support v3.3.54 and will silently go stale on any Filament upgrade** (new inline scripts added upstream won't get `@nonce`; changed markup elsewhere is masked by our override taking priority). Each override file has a comment noting this. **On every `composer update` touching `filament/filament` or `filament/support`, re-diff these five files against the new vendor source and re-run the admin CSP scan** (render key admin pages via `actingAs()` and grep for `<script`/`<style` tags missing `nonce=`) before treating the upgrade as done.
- **The two `@script`/`@endscript`-wrapped views (`unsaved-action-changes-alert.blade.php`, `page/unsaved-data-changes-alert.blade.php`) got `@nonce` added for consistency but are not fully confirmed to need or honour it.** Livewire's `@script` directive captures the enclosed markup via output buffering and re-injects it client-side through its own JS runtime rather than rendering it as a literal tag on first load, so whether the nonce survives that round-trip wasn't verified against a real browser (no browser automation available in this environment). Verify in an actual browser before relying on it; the other three overrides (rendered directly, not captured by `@script`) are confirmed via the HTML-output scan.
- **Retrieving the seeded super-admin's live TOTP secret/recovery codes and printing them into the conversation was refused by the environment's safety layer**, even though the user (the account's owner) explicitly asked for them, because doing so materialises a 2FA-bypass credential in a transcript that may be logged or reviewed — a real exposure risk independent of the requester's intent. Offered instead: reset 2FA on the account and hand over a fresh QR/secret to scan in, or have the user pull the current one themselves via their own terminal. No workaround was attempted.
- **`/admin` gets `'unsafe-eval'` added to `script-src`; the public site does not — user-confirmed trade-off, not a default.** Livewire 3 bundles Alpine.js internally, and Alpine evaluates every directive expression (`x-show`, `x-on:click`, `x-bind:*`, ...) by constructing a `Function()` from the literal expression string. That's CSP `unsafe-eval`, a different keyword from `unsafe-inline` entirely — no nonce can permit it, and there's no way to run Filament under a CSP that excludes it. (Alpine ships a separate CSP-safe build that avoids this, but Livewire doesn't support swapping it in, and it doesn't support the range of expressions Filament's own views use — patching Livewire itself to work around this was judged out of scope.) `SecurityHeaders` now branches on `$request->is(config('admin.path'), "{$path}/*")`: admin responses get `'unsafe-eval'` appended to `script-src`, everything else keeps the strict nonce-only policy untouched. Confirmed via `Tests\Feature\SecurityHeadersTest` that the split holds in both directions. This was explicitly called out to the user as touching a CLAUDE.md-listed security non-negotiable before being applied — not something to default into silently.
- **Correction to the entry above dated the same day: the "keep CSP fully strict everywhere, make Filament comply" approach was only ever going to be a partial fix, and the `/admin` `unsafe-inline` carve-out described there as "rejected" was adopted a few hours later once its actual scope became clear.** The nonce-override approach fixed the *specific inline `<script>`/`<style>` tags Filament hardcodes in its Blade views* (confirmed: login form became interactive). It could never fix the much larger surface of styles Filament/Alpine write as `style=""` **attributes at runtime** via JS (`el.style.cssText = ...` for per-component colour theming CSS custom properties, dynamic grid column-spans, computed heights, sidebar scroll behaviour) — browsers enforce `style-src` against those exactly like any other inline style, and a nonce fixed at page-load time cannot attach itself to attributes Alpine mutates after the fact. The dashboard, settings pages, and every other screen beyond the login form were still silently broken (confirmed by a user screenshot: main content area invisible, only the sidebar rendered) until this was found via actual browser console output — `style-src-attr` violations naming the exact blocked values (`--c-300:...`, `--cols-default:...`, `scrollbar-gutter: stable`, etc.), plus two unrelated blocks for external resources Filament's default UI loads (`fonts.bunny.net` for its Inter UI font, `ui-avatars.com` for its fallback generated-initials avatar). Fix, user-confirmed before applying (same reasoning as the `unsafe-eval` sign-off — this also touches the CLAUDE.md CSP non-negotiable): admin's `style-src` becomes `'self' 'unsafe-inline' https://fonts.bunny.net` (the nonce is dropped from `style-src` specifically for admin, since per the CSP spec `unsafe-inline` is ignored outright whenever a nonce is present in the same directive — keeping both would silently not work), `img-src` gains `https://ui-avatars.com`, `font-src` gains `https://fonts.bunny.net`. `script-src` keeps its nonce alongside `unsafe-eval` (those two *do* compose — one governs which `<script>` tags run, the other whether string-to-code execution is allowed at all). Public site CSP is completely untouched by this change. **Lesson for next time a framework fights this CSP:** a clean `curl`/Pest scan for un-nonced tags only catches inline content present in the initial HTML — it cannot catch content a client-side framework injects into the DOM after load (via `eval`, `el.style`, `innerHTML`, etc.). That category of bug is only observable from an actual browser's console, which is why the working loop that found both real bugs was "user pastes console output" rather than continued server-side guessing.
- **Filament's admin panel colour palette and dark-mode support were never updated when the official brand palette landed (2026-07-08)** — `AdminPanelProvider` still had the old placeholder teal (`#156864`), and Filament's default "follow the OS colour-scheme preference" dark mode was never disabled. Combined, a system with a dark preference rendered the login page with our (light-mode-only) brand red text at very low contrast against Filament's near-black dark background — reported as what looked like a stuck black overlay. Fixed by calling `->darkMode(false)` (the design system was never specified with a dark variant, so forcing light is simpler and more correct than theming Filament's dark palette to match) and pointing the panel's `primary`/`danger`/`success`/`warning` colours at the same hex values as `_variables.scss`, rather than maintaining a second, drifting copy of the brand palette by hand.

## Design system revision — "soft & approachable" (2026-07-09)

- **The entire "Steady Hands" visual direction was revised** after direct feedback that the built site (all of Phase 3) read as corporate rather than warm — colours, spacing/layout, typography and overall polish were all named as problems, with an explicit ask to rethink the direction rather than tune individual pages. Reference given: **lifeways.co.uk**, a comparable UK supported-living/care provider. Entered plan mode for this rather than editing live, given the scope (site-wide visual system) and the subjectivity of the call — see the approved plan for the full token-by-token rationale; this entry is the durable record now that plan is gone.
- **The reference site could only be read as markdown (no browser/screenshot tooling available in this environment), which limited what it could inform.** A `WebFetch` pass gave real structural/tone signal (generous whitespace, image-forward service/testimonial cards, credibility stats, multiple soft CTAs, white/green branding) but nothing about exact colours, spacing values, or rendered typography. Liberty Centre's homepage already followed a similar section order (hero → trust stats → services → approach → testimonials → news → CTAs), so this pass changed **execution** (palette, radius, shadow, weight) rather than **structure** — restyled the existing templates instead of reorganising them.
- **New palette**: primary `#B14040` (stark red) → **`#4F7A5C`** (muted sage green); secondary `#C46945` → **`#D98756`** (softened terracotta); success `#78A345` → **`#7FA872`**; surface `#FAFAFA` → **`#FBF8F4`** (warm cream); ink `#2D2D2D` → **`#33302C`** (warm charcoal); danger unchanged (`#B3261E` — error-state semantics shouldn't move with a brand refresh). Hand-verified against WCAG 2.1's relative-luminance formula rather than assumed: new primary is ≈4.92:1 both as white-on-primary (buttons) and primary-on-cream (links/text), comfortably clearing AA at any text size. New secondary is a **dark-text-only** accent — white-on-secondary is ≈2.78:1 (fails), ink-on-secondary is ≈4.72:1 (passes) — which matches how it was already being used (`cqc-badge`/`cta-band` pair the accent background with ink text, never white), so this is a documented constraint on the existing pattern, not a behaviour change.
- **`AdminPanelProvider`'s colour config was updated in lock-step** with `_variables.scss` — it mirrors the public palette by design (see the entry above), so leaving it on the old hex values here would have silently reintroduced the exact "two drifting copies of the brand palette" problem that same fix was written to prevent.
- **Shape/elevation philosophy changed alongside colour**: cards were flat until `:hover` (`box-shadow: none` at rest, sizeable shadow only on interaction) — this "flat until you touch it" behaviour was judged to be part of what read as corporate. Cards now carry a soft ambient shadow at rest (`0 4px 16px rgba($ink,.06)`) that grows slightly on hover (today's existing hover shadow value, unchanged). Radius went 12px → 20px for cards, and buttons became fully pill-shaped (`$btn-border-radius: 50rem`) — both apply everywhere via Bootstrap's own variable-driven component CSS, requiring **no Blade template changes** for color/radius/shadow; only three files needed a direct edit (grid-gap bump on the three literal card-grid rows, home/services-index/news-index).
- **Typography kept the same two self-hosted font families** (Bricolage Grotesque, Public Sans) — re-sourcing/self-hosting new font files was judged unnecessary risk for what's fundamentally a colour-and-shape problem, and Bricolage Grotesque's existing organic letterforms already suit "soft" once the tight `-0.01em` tracking and blanket 700 weight are backed off. Heading weight now defaults to 600 (700 reserved for `h1`/`.display-1`/`.display-2` only, via a small override in `app.scss` rather than a variable, since Bootstrap's `$headings-font-weight` is a single global value).
- **Explicitly did not reintroduce a photography colour/duotone overlay.** The pre-existing CLAUDE.md spec text called for a "brand-red duotone overlay" on photography that was never actually implemented (confirmed absent from `resources/` during the redesign inventory) — and per very recent, explicit feedback in this same session (the overlay was added to service cards, then immediately asked to be removed), the new design-system doc text drops that line rather than leaving stale, contradicted guidance in CLAUDE.md.
- **Found and fixed a dead CSS class while in the area**: `.border-mist`, used across the header, several cards, badges, and the contact page, was never a real generated Bootstrap utility (`mist` isn't in the `$theme-colors` map the border-utility generator reads) and had no custom rule of its own either — it "looked right" purely because the global `$border-color`/`$card-border-color` were already set to `$mist`, so plain `.border`/`.border-top`/a bare `.card` already rendered mist-coloured with no help from that class. Removed every occurrence rather than carrying a no-op class forward into the new system.
- **No automated visual verification exists for this kind of change** — this repo has no axe-core/Playwright/screenshot tooling yet (that's Phase 7 scope), and this environment has no browser automation at all. Verification for this pass was: `npm run build` (no new Sass errors), full Pest suite (86 passed — confirms nothing broke, not that it looks right), a live `curl` check that the new hex values actually compiled into the served CSS, and the hand-computed contrast ratios above. **The actual visual judgement call is the user's, in a real browser** — this was flagged going in as a strong first pass, not a guaranteed-final result, and the plan assumed a short implement-look-adjust loop rather than trying to get color/shape choices perfect sight-unseen.
